Diligent API Spec Request
Pre-Workshop — Action Required

This document contains 8 blocking questions that must be answered at or before the London workshop (w/c 30 March 2026). Each question maps directly to a delivery dependency — without answers, the Stealth Labs team cannot build the integration layer between our AI capabilities and the Diligent platform.

We have built our platform to be Diligent-native from day one. That means every use case reads from, writes to, and syncs with Diligent. But to do that, we need to know how Diligent exposes its data. These questions are not theoretical — they are the exact specs our engineers need to write code against.

Audience: Diligent Engineering & Product leadership. Deadline: Answers needed by end of London workshop (4 April 2026). Format: For each question, we need either API documentation, Swagger/OpenAPI specs, or a named technical contact who can walk us through the integration.

8 questions open
Workshop: 30 Mar – 4 Apr
Last updated: 19 Mar 2026
Question 1 of 8
Authentication & Token Federation
Stealth Labs operates as a separate application that needs to call Diligent APIs on behalf of authenticated users. Today our users log in via Auth0. We need to understand how to authenticate against Diligent’s APIs — whether that’s OAuth2 client credentials, a service account model, SAML federation, or a token exchange mechanism. This is foundational — nothing works without it.
How does Diligent authenticate third-party applications calling its APIs? Is there an OAuth2 token endpoint, a service account model, or a federation mechanism that would let our Auth0-issued tokens be exchanged for Diligent API tokens? What scopes or permissions model applies?
What we need back:
  • Auth flow documentation (OAuth2, API keys, or SAML)
  • Token endpoint URL and registration process for a new client application
  • Permission / scope model — can we request per-module access (e.g., Evidence only, ERM only)?
  • Rate limits and throttling policy
Blocks: All use cases
Question 2 of 8
Evidence Storage API
UC1 (Interview Co-Pilot) generates evidence during auditor interviews — transcripts, auto-drafted findings, and evidence links. UC2 (Control Testing) produces test results and exception reports. Both need to store evidence into Diligent and link it to the correct engagement, control, or finding. We cannot build the “last mile” of either use case without knowing how Diligent’s evidence store works.
Does Diligent expose an API for uploading evidence documents, linking evidence to engagements/controls/findings, and retrieving evidence metadata? What are the supported file types, size limits, and entity relationship model?
What we need back:
  • Upload endpoint — multipart file upload or pre-signed URL pattern?
  • Linking model — how do we associate an uploaded file with a specific engagement, control, or finding? Is it a foreign key, a tag, or a dedicated link endpoint?
  • Retrieval — can we list and download evidence by engagement/control?
  • File type restrictions and maximum file size
  • Versioning — does Diligent track evidence versions?
Blocks: UC1 Blocks: UC2
Question 3 of 8
Test Plan & Test Result API
UC2 (Automated Control Testing) runs agent-driven tests across the full evidence population for a given control. The agent produces structured pass/fail/exception results. These results need to be written back into Diligent’s control testing module so auditors see them in their normal workflow — not in a separate tool.
Does Diligent have APIs for reading test plans (the set of controls to be tested and their test procedures) and writing test results (pass/fail/exception per control, per evidence item)? Can we create new test runs programmatically?
What we need back:
  • Read: How to fetch test plans — list of controls, associated test procedures, evidence requirements
  • Write: How to submit a test result — pass/fail/exception status, supporting notes, linked evidence
  • Can we create a new “test run” or “test execution” programmatically, or must it be initiated in the Diligent UI?
  • Audit trail — does Diligent record who/what submitted the result? Can we tag it as “AI-assisted”?
  • Status sync — if we update a result, does it propagate to the control’s overall status?
Blocks: UC2
Question 4 of 8
ERM Risk Register API
UC3 (Automated Audit Reporting) needs to pull risk data from Diligent’s Enterprise Risk Management module to inform the Intelligence Scout and Risk Synthesizer agents. The report quality depends entirely on whether we can access the organisation’s current risk landscape — risk descriptions, ratings, owners, and linkages to controls and processes.
Does Diligent expose an API for reading the ERM risk register? Can we query risks by category, rating, owner, or linked control? What fields are available, and is the taxonomy configurable per tenant?
What we need back:
  • List/search endpoint for risks — filterable by category, rating, status
  • Risk entity schema — what fields exist (description, inherent/residual rating, owner, linked controls, linked processes)?
  • Is the risk taxonomy (categories, rating scales) configurable per tenant? If so, how do we discover the current configuration?
  • Can we read risk-to-control linkages?
Blocks: UC3
Question 5 of 8
Board Reporting / Boards API
UC3’s end product is a board-ready audit report. The Board Pack Curator agent needs to understand what format Diligent Boards expects — whether it’s a PDF upload, a structured data payload, or a native Diligent Boards document. Ideally, we deliver reports directly into the board portal so committee members see them without manual handoff.
Is there an API for publishing content to Diligent Boards? Can we programmatically create or update a board book / report pack? What format does it accept — structured JSON, PDF, or both?
What we need back:
  • Is there a “publish to Boards” API endpoint?
  • Accepted formats — PDF upload, structured data, or native Boards template?
  • Access model — who can publish? Is it restricted to specific roles?
  • If no API exists today, what is the recommended integration path? (e.g., shared drive, email, manual upload)
Blocks: UC3
Question 6 of 8
Compliance Framework Mapping API
The Intelligence Scout agent in UC3 needs to understand which regulatory frameworks apply to the organisation and how controls map to framework requirements. This is how the agent determines whether a control gap represents a regulatory risk versus an operational one. Without this, the report cannot distinguish between “nice to fix” and “regulator will flag this.”
Does Diligent maintain compliance framework mappings (e.g., SOX, ISO 27001, PCI-DSS mapped to controls)? Is there an API to read the framework library, the mapping between framework requirements and controls, and the compliance status per requirement?
What we need back:
  • List of supported frameworks and whether they are tenant-configurable
  • Mapping model — how are framework requirements linked to controls?
  • Compliance status per requirement — is there a calculated or manually set status?
  • Can we read the full mapping in bulk, or must we query per-control?
Blocks: UC3
Question 7 of 8
Third-Party Risk Management API
Modern audit reports must account for third-party risk — vendor dependencies, outsourced processes, and supply chain exposure. The Risk Synthesizer agent needs to pull third-party risk assessments from Diligent to flag where vendor risk compounds internal control weaknesses. This is increasingly what regulators ask about.
Does Diligent expose an API for its Third-Party Risk Management module? Can we read vendor risk assessments, due diligence status, and the linkage between vendors and internal controls or processes?
What we need back:
  • Vendor/third-party entity list with risk ratings
  • Due diligence and assessment status per vendor
  • Linkage model — how are vendors connected to controls, processes, or risk categories?
  • Is this module available in all Diligent tiers, or is it an add-on?
Blocks: UC3
Question 8 of 8
Vault / Document Management API
All three use cases generate documents that need to be stored persistently — interview transcripts (UC1), test result packs (UC2), and board-ready reports (UC3). If Diligent has a central document vault, we’d prefer to store directly there rather than maintaining a parallel document store. This also matters for data residency and retention compliance.
Does Diligent have a Vault or document management system with API access? Can we upload, tag, version, and retrieve documents programmatically? What are the data residency and retention policies?
What we need back:
  • Upload, retrieve, and delete endpoints
  • Tagging / metadata model — can we tag documents by use case, engagement, or control?
  • Versioning — does the vault track document versions?
  • Data residency — where is data stored? Is it tenant-configurable (EU, UK, US)?
  • Retention policies — automatic expiry or manual management?
  • Access control — can we set per-document permissions?
